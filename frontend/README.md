# Frontend

> Part of the [Campus Event App](../README.md) monorepo — see the [root README](../README.md) for the project overview, and [`../backend`](../backend/README.md) for the Dart Frog API.

Flutter client for the Campus Event App. Talks to the Dart Frog backend (see `../backend`) — see the [Setup](#setup) section below to get running.

> New to Flutter? Start with [Learn Flutter](https://docs.flutter.dev/get-started/learn-flutter), the [first-app codelab](https://docs.flutter.dev/get-started/codelab), and the [online docs](https://docs.flutter.dev/).

---

## Tech Stack

- **Framework:** Flutter
- **State Management:** Provider (`ChangeNotifier`)
- **Auth:** Firebase Authentication (email/password) — used only for the custom-token sign-in exchange
- **Data:** everything goes through the Dart Frog REST API (see `backend/`), hosted on Google Cloud Run. The app never talks to Firestore or Cloudinary directly
- **Platform:** Android is the primary target; a web scaffold exists but isn't a shipped target

## Architecture

This app follows a **features-based architecture**. Each feature owns its own data, state, and UI — features should not directly import internals from other features.

```
lib/
├── main.dart                      # MaterialApp.router, top-level MultiProvider, Firebase init
├── firebase_options.dart          # generated by `flutterfire configure`
├── core/
│   ├── constants/                 # role strings, error codes, API base URL, branch indices
│   ├── network/                   # api_client.dart, api_exception.dart
│   ├── models/                    # shared models (User, ApiResponse, campus map)
│   ├── router/                    # app_router.dart — go_router config + redirect auth guard
│   ├── theme/                     # app_theme.dart
│   ├── fonts/                     # bundled font files (Inter)
│   └── utils/                     # validators.dart
├── features/
│   ├── auth/
│   │   ├── data/                  # auth_repository.dart, firebase_auth_service.dart
│   │   ├── providers/             # auth_provider.dart — ChangeNotifier
│   │   └── presentation/          # sign in/up, forgot password
│   ├── events/
│   │   ├── data/                  # event_repository.dart, campus_map_data.dart
│   │   ├── models/                # event.dart, campus_map.dart
│   │   ├── providers/             # event/list/dashboard/calendar/create/created providers
│   │   └── presentation/
│   │       ├── screens/           # dashboard, events, calendar, create/edit, created, previous, map
│   │       └── widgets/           # event card, event modal, calendar grids, floor map view
│   ├── admin/
│   │   ├── data/                  # user_management + event_moderation repositories
│   │   ├── models/                # managed_user.dart, pending_event.dart
│   │   ├── providers/             # role_promotion + event_approval providers
│   │   └── presentation/          # admin landing, role management, review queue
│   └── profile/
│       ├── providers/             # settings_provider.dart
│       └── presentation/          # edit profile, settings
└── shared/
    └── widgets/                   # main shell + navbar, header, dialogs, cards, role tag,
                                   # tab-focus refresh helpers reused app-wide
```

**Data flow convention:**
`Screen/Widget → Provider (ChangeNotifier) → Repository → Backend API`

UI never calls a repository directly — always go through the provider so state changes trigger rebuilds correctly.

> **State management note:** the app is built on Provider (`ChangeNotifier`). Riverpod (`flutter_riverpod` 2.x) is also wired in and available for newer work — the two coexist, so run project lints with `dart run custom_lint`.

> **Navigation:** all routing goes through `go_router` (config in `core/router/app_router.dart`). The bottom-nav tabs are a `StatefulShellRoute.indexedStack`; add new screens as `GoRoute`s with a `Routes` constant and navigate via `context.push`/`go`/`pop`, not `Navigator` + `MaterialPageRoute`.

**Provider registration:** all top-level providers are registered once in `main.dart` (`MyApp`) inside a single `MultiProvider`. When adding a new feature provider, add it there — don't create a second registration point.

## Setup

### Prerequisites

- Flutter SDK (stable channel)
- Android Studio or VS Code with Flutter/Dart plugins
- An Android emulator or physical device with USB debugging enabled
- Access to the project's Firebase console (ask the team lead)

### 1. Clone and install dependencies

```bash
git clone <repo-url>
cd frontend
flutter pub get
```

### 2. Firebase setup

1. Get `google-services.json` from the team lead (or download it yourself from the Firebase console if you have access).
2. Place it at:
   ```
   android/app/google-services.json
   ```
3. This file is **gitignored** — never commit it. Each developer needs their own copy locally.

### 3. Backend API URL

The app talks to the Dart Frog backend hosted on Cloud Run. Set the base URL in:

```
lib/core/constants/api_constants.dart
```

```dart
const String apiBaseUrl = 'https://TODO-cloud-run-url.a.run.app';
```

> **Note:** The actual Cloud Run URL is TBD and will be shared once the backend is deployed. Use the placeholder above until then. During local backend development, point this to your backend dev's local Dart Frog server instead (e.g. `http://<their-local-ip>:8080`) — `localhost` won't work since the team is fully remote.

### 4. Run the app

```bash
flutter run
```

## Conventions

- **State management:** Provider (`ChangeNotifier` + `MultiProvider`). Keep business/state logic in providers, not widgets.
- **API calls:** only from `data/` repository files, never directly from widgets or providers.
- **Error handling:** backend returns a standardized `{ success, message, ... }` shape — surface `message` to the user via shared error widgets in `shared/widgets/`, don't write one-off error UI per screen.
- **Pull-to-refresh:** used for event feeds (not real-time listeners). Notifications are fetched on demand, not pushed.
- **Soft deletes:** the backend never hard-deletes records. UI should treat "deactivated" users/events as hidden, not gone — don't assume a 404 means permanently deleted.
